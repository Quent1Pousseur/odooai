## BLOC 5 — Engineering Data & Odoo

### Data Engineer (11)
- **Challenge sur le stockage** : "Le plan prevoit des fichiers JSON dans `knowledge_store/`. Pour 10 modules, ca passe. Pour 200+ modules (tout Odoo), c'est 2000+ fichiers JSON. La recherche sera lente. On devrait prevoir un index SQLite ou un JSONB PostgreSQL des le depart, pas apres."
- **Challenge** : "Le `.gitignore` exclut les JSON du knowledge_store. Ca veut dire que les Knowledge Graphs ne sont pas versionnes. Si on regenere avec un bug dans le parseur, on perd les bons. Il faut un mecanisme de snapshot/backup."

### Odoo Expert (10)
- **Challenges critiques sur DATA-001** :
  1. "Le schema `ModelDefinition` a `inherit: list[str]`. Mais Odoo a DEUX types d'heritage : `_inherit` (extension) et `_inherits` (delegation via Many2one). Le parseur DOIT les distinguer sinon l'intelligence business sera fausse."
  2. "`_inherit` sans `_name` signifie 'j'etends un modele existant, pas j'en cree un nouveau'. Le parseur doit fusionner les champs, pas creer un nouveau modele."
  3. "Les champs `compute` en Odoo peuvent etre `store=True` (persistes) ou `store=False` (calcules a la volee). Le schema le prevoit, mais le parseur doit correctement detecter le `store` kwarg — par defaut il est `False` pour les computed."
  4. "Les `related` fields sont un raccourci (`related='partner_id.name'`). Ils ne sont pas dans le schema actuel. C'est utilise massivement dans Odoo."
  5. "Les vues XML utilisent `inherit_id` pour etendre des vues existantes. Le parseur XML doit-il fusionner les vues heritees ou juste lister les extensions ?"

### Data Scientist (28)
- **Challenge** : "Les Knowledge Graphs vont etre les donnees d'entrainement de nos BA Profiles. Si les donnees sont biaisees (modules bien parses vs modules mal parses), les profiles seront biaises. Il faut une metrique de completude par module : % champs extraits vs % champs reels. Objectif : >95% de precision."
