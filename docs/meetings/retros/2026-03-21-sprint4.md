## Ce qui a MOINS BIEN fonctionne

### 1. La regle zero inactivite oubliee 4 fois (Fondateur)
**Fondateur** : "J'ai du rappeler 4 fois que tout le monde doit travailler ou se former. C'est inacceptable. La regle est maintenant dans le MANIFESTO, le CLAUDE.md, et la memoire. Il n'y a plus d'excuse."

**Action** : Template daily checklist obligatoire. AUCUNE derogation.

### 2. Les 4 issues tests reportees (QA Automation)
**QA Automation (48)** : "Les tests E2E et integration sont reportes a Sprint 5. On n'a pas de tests end-to-end dans le CI. C'est un risque."

**Action** : Priorite #1 Sprint 5.

### 3. Pas de code ce sprint (CTO)
**CTO (02)** : "On a ferme 48 issues mais seulement ~5 ont implique du code (edge case tests, prompt fix, design pass, Dockerfile). Le reste c'est de la doc. C'est bien pour la stabilisation mais Sprint 5 doit etre un sprint de code."

**Action** : Sprint 5 = 60% code, 40% docs/business.

### 4. Le rapport d'activite incomplet (PM)
**PM (04)** : "Certains dailys n'avaient pas le tableau 48 agents. C'est mon job de m'assurer que tout le monde est trace."

**Action** : Template daily avec tableau 48 agents BLOQUANT.

### 5. Trop de docs, pas assez de tests terrain (CPO)
**CPO (03)** : "On a 26 learning CRs excellents mais 0 beta user reel. Toute cette theorie doit etre validee par le terrain. Sprint 5 = premiers vrais utilisateurs."

## Lessons learned

1. **Le feature freeze fonctionne** — a refaire quand on accumule trop de dette
2. **Les learning CRs sont un accelerateur** — pas du temps perdu, du temps investi
3. **Le wellbeing se mesure** — 5.9 → 7.0, c'est concret et ca impacte la productivite
4. **La competitive intel clarifie le positionnement** — le marche est vide, speed matters
5. **La regle zero inactivite doit etre AUTOMATIQUE** — pas un rappel du fondateur
