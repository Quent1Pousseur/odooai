## Ce qui n'a pas marche ❌

### 1. Pas de plan mode systematique pour CORE-002 — CTO (02)
Pour CORE-001 on a bien fait : plan complet, phases A-G, ordre clair. Pour CORE-002, le plan etait plus leger. Ca a marche car on avait la reference prodooctivity, mais pour des specs plus complexes (SEC-001, DATA-001) il faudra etre plus rigoureux.

### 2. Pas de tests d'integration — QA Lead (13)
Tous les tests sont des tests unitaires avec mocks. On n'a aucun test qui parle a un vrai Odoo. C'est normal pour le jour 1, mais ca doit etre adresse rapidement. Sans tests d'integration, on ne sait pas si le client fonctionne reellement.

### 3. Le stub pattern cree de la dette — Backend Architect (08)
On a des stubs partout (AnthropicProvider, DB, Redis reel, Guardian pipeline). C'est voulu pour avancer vite, mais chaque stub est une promesse non-tenue. Il faut tracker ca et ne pas laisser les stubs s'accumuler trop longtemps.

### 4. Config incomplete — Infra Engineer (12)
Le `config.py` a des valeurs par defaut vides (`odoo_url: str = ""`, `odoo_crypto_key: str = ""`). En production, ca va creer des erreurs silencieuses si quelqu'un oublie de configurer. Il manque une validation fail-fast au demarrage.

### 5. Pas de CI/CD encore — DevOps (22)
On fait `make check` manuellement. Si quelqu'un oublie, du code non-conforme pourrait passer. GitHub Actions devrait etre la prochaine priorite pour automatiser ca.

## Ce qu'on peut ameliorer 🔧

### 1. Validation config au demarrage — Backend Architect (08)
Ajouter une validation dans `create_app()` ou `lifespan()` qui verifie que les settings critiques sont non-vides en production. Pattern : fail-fast, pas fail-silent.

**Action** : Ajouter dans ODAI-CORE-003 une methode `Settings.validate_production()` qui raise `ConfigurationError` si des secrets manquent en prod.

### 2. Instance Odoo de test — Odoo Expert (10)
Pour les tests d'integration, on a besoin d'un Odoo de test (docker ou SaaS de dev). Ca permettrait de valider le client contre une vraie instance.

**Action** : Ajouter un `docker-compose.test.yml` avec un Odoo minimal pour les tests d'integration.

### 3. Tracker les stubs — PM (04)
Chaque stub devrait etre dans le todo avec la spec qui le remplit. Si un stub reste plus de 2 sprints, c'est un signal.

**Action** : Maintenir une section "Stubs actifs" dans tasks/todo.md.

Ne pas attendre — GitHub Actions est independant des autres specs et peut avancer en parallele.

**Action** : ODAI-INFRA-001 en priorite P1, livrable cette semaine.

### 5. Logs structures des le depart — Observability (38)
On utilise `structlog` mais on n'a pas encore configure le formatage (JSON en prod, console en dev). Il faudrait le faire dans ODAI-CORE-003.

**Action** : Ajouter la config structlog dans ODAI-CORE-003.

### 6. Le port IOdooClient pourrait avoir read_group et name_search — Odoo Expert (10)
On les a exclus du MVP, mais `read_group` est essentiel pour les rapports et dashboards. `name_search` est utilise partout dans l'UI Odoo. Il faudrait les ajouter rapidement.

En tant que non-technique, je peux comprendre la structure : `_errors.py` pour les erreurs, `_json2.py` pour le protocole moderne, `_xmlrpc.py` pour l'ancien. C'est lisible.

## Lecons a Ajouter dans lessons/

### Lecon 1 : Split > Monolithe
Le client Odoo de prodooctivity (1553 lignes, 1 fichier) est fonctionnel mais difficile a naviguer. Le split en 6 fichiers avec responsabilite unique est nettement superieur pour la lisibilite et la maintenance. **Regle : si un fichier depasse 200 lignes, chercher un split naturel.**

### Lecon 2 : Reference ≠ Copie
Etudier une reference en profondeur avant de coder est essentiel, mais copier sans comprendre est dangereux. On a ameliore le error handling, ajoute le rejet portal, et simplifie la signature. **Regle : toujours se demander "comment faire mieux ?" en lisant du code de reference.**

### Lecon 3 : mypy --strict des le jour 1
Ajouter mypy --strict apres coup est douloureux. L'avoir des le depart force des decisions de typage propres. Le cout marginal est faible quand on code correctement. **Regle : jamais de regression mypy — c'est un cliquet.**
