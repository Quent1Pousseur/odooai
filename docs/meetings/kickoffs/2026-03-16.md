**Quentin Pousseur — Fondateur**

Bienvenue a tous. Aujourd'hui on lance OdooAI.

Le probleme est simple : les PME n'utilisent Odoo qu'a 10-30% de ses capacites. Les consultants coutent 150-300€/h, la documentation est massive, et personne n'a construit l'outil qui transforme la connaissance contenue dans le code source d'Odoo en intelligence business accessible.

Points cles a retenir :
- OdooAI est un **concurrent** de prodooctivity, pas un fork. On construit from scratch, meilleur du marche
- Notre moat : les **Knowledge Graphs** extraits du code source. Pas de la doc, pas de ChatGPT
- **SaaS first**, closed source, architecture LLM-agnostic
- Le code source Odoo est fourni manuellement par moi. Pas de scraping, pas de clone
- On vise le marche mondial : 180+ pays, des centaines de milliers de PME sous-exploitant Odoo

Notre objectif a 24 mois : etre le leader reconnu de l'assistance IA pour Odoo.

Merci Quentin. Je vais poser le cadre business.

**Marche cible** : les PME qui utilisent Odoo et qui n'ont pas les moyens ou le temps de payer un consultant. C'est un marche enorme et sous-servi.

**Positionnement** : on ne remplace pas les consultants — on democratise ce qu'ils savent. Un consultant facture €150-300/h pour de la configuration. Nous, on propose la meme intelligence a €49-399/mois, 24h/24, dans la langue du client.

**Priorites Phase 1** :

**Ce que je surveillerai** : chaque decision sera passee au filtre "est-ce que ca aide Marie, la dirigeante de PME qui n'a pas le temps de lire la doc Odoo ?" Si la reponse est non, on ne le fait pas.

### CTO (02) — Architecture Technique

La stack est definie et non-negociable pour Phase 1 :
- **Backend** : Python 3.11+, FastAPI, SQLAlchemy async, Pydantic v2
- **LLM** : Anthropic Claude (Haiku/Sonnet/Opus selon le type de tache)
- **DB** : SQLite en dev, PostgreSQL en prod
- **Cache** : Redis
- **Quality** : ruff, mypy --strict, bandit, pytest

**Principes architecturaux** :
1. **No magic** — du code lisible, pas de meta-programmation
2. **Immutability by default** — frozen dataclasses pour les value objects
3. **Fail-fast** — les erreurs eclatent tot, pas en silence
4. **LLM-agnostic** — interface abstraite, on peut switcher de provider en 1 jour
5. **Observability** — structlog partout, chaque requete tracable

**Structure du code** :
```
odooai/
  main.py              # FastAPI entrypoint
  config.py            # Settings (pydantic-settings)
  domain/              # Value objects, entities, enums
  services/            # Business logic
  agents/              # LLM agent implementations (produit)
  connectors/          # Odoo XML-RPC/JSON-RPC
  security/            # Guardian, anonymization, encryption
  api/                 # FastAPI routers
```

Le scaffold est deja en place. `make run` demarre le serveur, `make check` lance tous les controles qualite.

**Alerte** : aucune ligne de code ne sera ecrite avant d'avoir une spec approuvee. C'est la regle.

## PARTIE 3 — Culture & Processus

### PM (04) — Presentation du Cadre de Travail

Je vais presenter les regles du jeu. Elles ne sont pas negociables.

**5 principes du MANIFESTO :**

1. **Excellence avant la vitesse** — Un bug en production coute 100x plus qu'un jour de retard. On ne coupe jamais les coins ronds.

2. **Le challenge est un devoir** — Chaque agent DOIT questionner le travail des autres. Respectueusement, avec des arguments et des alternatives. Le silence face a un probleme est la pire forme d'irresponsabilite.

3. **Transparence totale** — Pas de silos. Les problemes sont escalades immediatement, jamais caches. Les erreurs sont admises, pas dissimulees.

4. **La securite est sacree** — On manipule des donnees d'entreprise : salaires, factures, contrats. Une fuite = fin de la confiance.

5. **Le client est la raison d'etre** — Chaque decision : "est-ce que ca aide Marie ?"

**Droit de VETO** :
- Tout agent peut bloquer une decision dans son domaine d'expertise
- Un VETO n'est pas un blocage — c'est un signal que quelque chose merite plus d'attention
- Procedure : travail s'arrete → les parties presentent leurs arguments → le fondateur tranche → decision documentee

**Regle des 2 reviews** :
- Code : Senior Dev + architecte du domaine
- Specs : PM + expert du domaine
- Prompts : Prompt Engineer + Odoo Expert
- Securite : Security Architect + un second reviewer

**Workflow de developpement** :
```
SPEC → REVIEW → CODE → REVIEW → TEST → MERGE
```
- Rien ne se code sans spec approuvee
- Format spec : ODAI-[DOMAINE]-[NUMERO]
- Commit : `[ODAI-XXX] type: description`
- Squash merge only, 2 approvals minimum, CI verte obligatoire

**Daily Standup** — tous les jours, 45 minutes :
- Tour de table en 8 blocs (Direction → Business → Design → Backend → Data → Infra → Security → Delivery)
- 2 min max par agent : fait hier, fait aujourd'hui, blocages
- 10 min entraide, 5 min signaux globaux
- Notes documentees le jour meme dans `meetings/daily/`
