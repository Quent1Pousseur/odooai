## PARTIE 3 — Challenges techniques

### Odoo Expert (10)
- **VETO leve** de Semaine 1 (schemas KG OK). Nouveau challenge :
- "Avant de generer des BA Profiles, je DOIS valider la qualite des KG. Je veux comparer le KG `sale` genere par notre parseur vs ce que je sais du vrai module. Si le parseur rate des champs importants, les BA Profiles seront faux."
- **Proposition** : "Je fais le quality check AVANT que l'AI Engineer lance la BA Factory."

> **QA Lead (13) soutient** : "Ca devrait etre un test automatise : on definit les champs attendus pour `sale.order` et on verifie que le KG les contient."

### AI Engineer (09)
- "Pour le BA Profile Generator, ma strategie :
  1. Charger le KG d'un module (ex: `sale`)
  2. Extraire les 20 champs les plus importants par modele (field_scorer)
  3. Construire un prompt : 'Voici le schema du module sale. Genere un profil d'analyste business.'
  4. Envoyer a Sonnet, recevoir le BA Profile
  5. L'Odoo Expert valide le resultat"
- **Challenge** : "Il faut definir le format exact du BA Profile AVANT de coder. Sinon on genere du texte libre inexploitable."

> **Prompt Engineer (25) intervient** : "Je co-redige le prompt avec l'AI Engineer. Et j'ai besoin des criteres d'evaluation : precision factuelle >95%, completude >80%, exploitabilite LLM score 1-5."
>
> **Data Scientist (28) intervient** : "Comment on mesure 95% de precision factuelle ? Il faut un ground truth — c'est le role de l'Odoo Expert."

### Security Architect (07)
- "Le BA Profile Generator va envoyer des KG au LLM. Les KG contiennent des noms de modeles, de champs, de contraintes. Ce n'est PAS des donnees client, c'est de la meta-information Odoo. Mais les help texts extraits du code source pourraient contenir du prompt injection. Il faut sanitizer avant d'envoyer."

> **AI Safety (33) soutient** : "D'accord. Sanitizer les strings extraites du code source avant de les inclure dans un prompt. Supprimer les caracteres de controle, les balises HTML, et les patterns d'injection."

### Senior Backend Dev (19)
- **Challenge** : "L'Orchestrator va etre le composant le plus complexe. Il faut qu'il soit bien decoupe des le depart. Je propose :
  - `odooai/agents/orchestrator.py` — routing
  - `odooai/agents/ba_agent.py` — consultation BA Profile
  - `odooai/agents/chat.py` — CLI loop
  Pas un monolithe."

### DBA (30)
- "Les BA Profiles generes doivent etre stockes quelque part. Fichiers JSON comme les KG ? Ou en DB directement ? Les KG sont en fichiers parce qu'ils sont generes offline. Les BA Profiles aussi — donc fichiers JSON, meme pattern que `knowledge_store/`."
